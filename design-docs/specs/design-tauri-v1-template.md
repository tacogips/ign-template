# Tauri V1 Template Design

**Status**: Completed

## Spec Reference

- Workflow input: Create `tauri-v1` ign template from the local `chilla` reference application
- Reference template roots: `bun-ts-v1`, `rust-v1`
- Reference application: sibling `chilla` checkout used as a structural reference
- Verification: `ign template update tauri-v1`; `ign template check tauri-v1`

## Problem

Create a new ign template root named `tauri-v1` for a reusable Tauri desktop application scaffold. The template must follow existing ign-template conventions, use `chilla` as a structural reference, expose appropriate ign variables, update the repository README, and remain valid under ign template update/check.

## Boundaries

The template is a generic scaffold, not a copy of the chilla product. It should preserve the useful project shape from chilla:

- Bun, Vite, TypeScript frontend.
- Tauri backend under `src-tauri/`.
- Rust workspace with a backend crate and `tauri.conf.json`.
- Taskfile commands for development, build, test, check, lint, and clean.
- Nix development environment that includes Bun, Rust tooling, Tauri build dependencies, and ign-friendly project setup.

The template should not include chilla-specific Markdown viewer, Git viewer, release signing, notarization, custom window chrome, file watcher, media streaming, or product-specific commands unless they are needed for a minimal Tauri scaffold.

## Template Variables

`tauri-v1/ign-template.json` should be generated by `ign template update tauri-v1`, not hand-edited for hash changes. The implementation should include variable usages that cause ign to collect at least these variables:

| Variable | Type | Default | Purpose |
|----------|------|---------|---------|
| `PROJECT_NAME` | string | `{current_dir}` | npm package name, generated directory identity, and app-facing default name. |
| `VERSION` | string | `0.1.0` | npm, Cargo, and Tauri config version. |
| `DESCRIPTION` | string | `A Tauri desktop app` | package, Cargo, flake, and README description. |
| `REPOSITORY` | string | `https://github.com/user/repo` | package metadata and generated docs. |
| `HOMEPAGE` | string | `https://github.com/user/repo` | package metadata and generated docs. |
| `AUTHOR_NAME` | string | empty | Optional author metadata. |
| `AUTHOR_EMAIL` | string | empty | Optional author metadata. |
| `HAS_AUTHOR` | bool | required | Controls author metadata blocks. |
| `TAURI_PRODUCT_NAME` | string | `Tauri App` | User-visible Tauri product/window name. |
| `TAURI_IDENTIFIER` | string | `com.example.app` | Tauri bundle identifier. |
| `RUST_CRATE_NAME` | string | `app` | Rust package name after user supplies a crate-safe value. |
| `RUST_LIB_NAME` | string | `app_lib` | Rust library target name after user supplies an underscore-safe value. |
| `RUST_VERSION` | string | `1.92.0` | Rust toolchain version, aligned with chilla because current Tauri dependency resolution requires newer Cargo support than Rust 1.83. |
| `EDITION` | int | `2021` | Cargo edition. |

Because ign does not derive sanitized variables, `RUST_CRATE_NAME`, `RUST_LIB_NAME`, and `TAURI_IDENTIFIER` must be separate user-provided variables instead of assuming `PROJECT_NAME` is valid for every target format. Defaults must be literal values and must not embed another `@ign-var:*@` directive inside the default expression. `HAS_AUTHOR` is the only conditional driver and must remain `bool`; string variables must not be used in `@ign-if` conditionals.

## Generated Project Shape

The template should generate a self-contained project similar to:

```text
.
├── AGENTS.md
├── Cargo.toml
├── Taskfile.yml
├── flake.nix
├── index.html
├── package.json
├── src/
│   ├── main.ts
│   ├── style.css
│   └── vite-env.d.ts
├── src-tauri/
│   ├── Cargo.toml
│   ├── build.rs
│   ├── capabilities/default.json
│   ├── src/lib.rs
│   ├── src/main.rs
│   └── tauri.conf.json
├── tsconfig.json
└── vite.config.ts
```

Optional local workflow folders such as `design-docs/` and `impl-plans/` may be included only if they match the conventions already present in `bun-ts-v1` and `rust-v1` and do not introduce product-specific chilla content.

## Reference Mapping

| Reference | Use | Design Decision |
|-----------|-----|-----------------|
| `bun-ts-v1/package.json` | Bun scripts and metadata variable style. | Reuse concise scripts and metadata conventions. |
| `bun-ts-v1/flake.nix` | Bun-focused Nix development shell. | Reuse simple dev-shell style where possible. |
| `rust-v1/Cargo.toml` | Rust metadata, author conditionals, edition/toolchain variables. | Reuse Cargo variable conventions and separate Rust toolchain variable. |
| `rust-v1/Taskfile.yml` | Task runner conventions. | Keep task names predictable: `default`, `build`, `test`, `check`, `fmt`, `fmt-check`, `lint`, `clean`. |
| `chilla/package.json` | Tauri, Bun, Vite, TypeScript script shape. | Use Tauri/Bun/Vite structure but avoid chilla-only dependencies. |
| `chilla/src-tauri/Cargo.toml` | Tauri v2 Cargo layout. | Use Tauri v2 crate layout and minimal dependencies. |
| `chilla/src-tauri/tauri.conf.json` | Tauri build config, dev URL, frontend dist. | Reuse build flow and dev port shape; avoid chilla-specific window/security choices unless needed. |
| `chilla/vite.config.ts` | Vite Tauri dev host behavior. | Reuse Tauri dev server port `1420`, strict port, HMR host override, and ignored `src-tauri` watch path. |

## README Update

Repository `README.md` should list `tauri-v1` in "Available Templates" and include checkout examples for:

- GitHub checkout without output directory.
- GitHub checkout with output directory.
- Dry-run preview.

README changes should keep existing templates visible and avoid broad unrelated documentation cleanup.

## Validation Rules

Implementation must run:

```bash
ign template update tauri-v1
ign template check tauri-v1
```

If either command fails because ign behaves unexpectedly, the implementation must not silently work around the failure. The failure should be reported with the command, expected behavior, actual behavior, and output so an issue can be filed for `https://github.com/tacogips/ign/issues`.

After validation, review the diff for `tauri-v1/ign-template.json` and confirm the hash and variables changed only as expected.

## Intentional Divergences From Chilla

- Template name `tauri-v1` means version 1 of the ign template, while the framework baseline follows chilla's Tauri v2 structure.
- The generated app should be a minimal reusable Tauri scaffold, not a Markdown/Git viewer.
- Release signing, notarization, bundle DMG tasks, e2e driver scripts, and product-specific runtime wrappers should be omitted from the initial scaffold.
- Tauri identifier and Rust crate naming are modeled as explicit variables because ign does not sanitize `PROJECT_NAME` for those formats.

## Risks

- Tauri, npm package, Cargo package, Rust lib target, and bundle identifier naming rules differ; invalid defaults can produce generated projects that fail to build.
- Copying chilla dependencies too broadly can make the template heavy and product-specific.
- Nix GUI dependencies for Linux and Darwin can be complex; the first implementation should prefer a development shell that supports local build/check commands before adding full reproducible package outputs.
- Rust 1.83 is too old for current Tauri dependency resolution because transitive crates may require Cargo/Rust 2024 support; the template therefore follows chilla's Rust 1.92.0 default.
- README currently omits some existing template roots; the tauri-v1 README update should stay focused while not removing existing listed templates.

## Open Questions

No user-blocking decisions remain for the design step. The implementation should proceed with `tauri-v1` as the ign template version and Tauri v2 as the framework baseline, based on the supplied chilla reference.
